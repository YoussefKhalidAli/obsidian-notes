## 1. What is a Load Balancer?

- A **Load Balancer** is a service that:
  - Receives incoming traffic  
  - **Distributes it across multiple backend servers (EC2 instances)**  

- Provides a **single entry point (endpoint)** for users  

---

## 2. How It Works

- Users send requests to the Load Balancer  
- Load Balancer forwards requests to different EC2 instances  
- Traffic is distributed to avoid overloading a single instance  
- Users do **not know which instance handles their request**  

---

## 3. Why Use a Load Balancer?

### Benefits

1. **Single point of access**
   - One DNS instead of multiple servers  
1. **Distributes traffic**
   - Prevents overload  
1. **High Availability**
   - Works across multiple AZs  
1. **Fault tolerance**
   - Stops sending traffic to unhealthy instances  
1. **Health Checks**
   - Detects failing instances automatically  
1. **SSL Termination**
   - Handles HTTPS encryption  
1. **Stickiness (Sessions)**
   - Routes user to same instance using cookies  
1. **Security**
   - Separates public access from backend servers

## 5. Sticky Sessions
- Sticky sessions, also called **session affinity**, ensure that a client is always routed to the **same backend instance** for multiple requests.
- Normally, load balancers distribute traffic across all instances evenly.
- Stickiness is useful for preserving **user session data** (e.g., login sessions).

## 5.1 How Sticky Sessions Work
1. A **cookie** is generated by the load balancer or the application.
2. The cookie stores information about which backend instance handled the initial request.
3. Subsequent requests from the same client use the cookie to route to the **same backend instance**.
4. Sticky sessions can introduce **load imbalance** if some users are "stickier" than others.

## 5.2 Cookie Types
### 1. Application-Based Cookie
- Generated by the **application** itself.
- Customizable attributes.
- Must be specified individually for each target group.
- Reserved names not allowed: `AWSALB`, `AWSALBAPP`, `AWSALBTG`.
- `AWSALBAPP` is an application cookie generated by the load balancer itself
### 2. Duration-Based Cookie
- Generated by the **load balancer**.
- Name:
  - `AWSALB` for ALB
  - `AWSELB` for Classic Load Balancer
- Expiration based on a specified duration (e.g., 1 second to 7 days).

## 6. Cross-Zone Load Balancing
- Distributes traffic **evenly across ALL instances in ALL Availability Zones (AZs)**.
- Without it → traffic is only distributed **within the same AZ**.
- Helps solve **uneven instance distribution across AZs**.

## With Cross-Zone Load Balancing (ENABLED)
- Each load balancer node:
  - Sends traffic to **ALL  instances in it's target group across all AZs**
- Result:
  - **Even traffic distribution across all instances**
  - No AZ-level imbalance
### Example:
- AZ1 → 2 instances  
- AZ2 → 8 instances  
- Total = 10 instances  
➡ Each instance gets **~10% traffic**, regardless of AZ  

## Without Cross-Zone Load Balancing (DISABLED)

- Each load balancer node:
  - Sends traffic **ONLY to instances in its own AZ**
- Result:
  - Traffic split per AZ, not per instance
  - Can cause **imbalance**

### Example:
- AZ1 → 2 instances → each gets **25% traffic**
- AZ2 → 8 instances → each gets **6.25% traffic**

➡ Uneven load → possible overload in smaller AZ

## 6.1 Behavior by Load Balancer Type

| Load Balancer | Default | Can Disable? | Cost |
|--------------|--------|-------------|------|
| **ALB** (Application) | Enabled | Yes (per target group) | ❌ No inter-AZ cost |
| **NLB** (Network)     | Disabled | Yes | ✅ Inter-AZ charges apply |
| **GWLB** (Gateway)    | Disabled | Yes | ✅ Inter-AZ charges apply |
| **CLB** (Classic)     | Disabled | Yes | ❌ No inter-AZ cost |

## 7. SSL / TLS
- **SSL (Secure Sockets Layer)** → Old term  
- **TLS (Transport Layer Security)** → New, modern version  
- Both are used for **encrypting data in transit (in-flight encryption)**  
➡ Ensures:
- Data is **secure between client and server**
- Prevents **eavesdropping / data theft**
## 7.1 SSL Certificates
- Used to enable **HTTPS (secure HTTP)**
- Issued by **Certificate Authorities (CA)**:
  - Examples: Let's Encrypt, DigiCert, GlobalSign
- Have:
  - **Expiration date** → must be renewed
  - Indicated by 🔒 in browser

## 7.2 SSL in Load Balancers
### SSL Termination
- Client → Load Balancer → EC2
Flow:
1. Client connects using **HTTPS (encrypted)**
2. Load Balancer **decrypts traffic** (SSL termination)
3. LB → EC2 can be:
   - HTTP (unencrypted, inside VPC)
   - HTTPS (optional, more secure)

## 7.3 SNI (Server Name Indication)
### Problem:
- One server → multiple websites → multiple SSL certs needed
### Solution:
- Client specifies **hostname during TLS handshake**
- Server selects **correct certificate**

### Example:
- `www.site1.com`
- `api.site2.com`

➡ Same Load Balancer, different certs

Client says:
> "I want site1.com"

LB:
> Uses **site1 certificate**

## 7.4 Load Balancer Support for SSL & SNI

| Load Balancer | SSL Support | Multiple Certificates | SNI Support |
|--------------|------------|----------------------|-------------|
| **CLB**      | Yes        | ❌ No (1 cert only)  | ❌ No       |
| **ALB**      | Yes        | ✅ Yes               | ✅ Yes      |
| **NLB**      | Yes        | ✅ Yes               | ✅ Yes      |

## 8. Connection Draining / Deregistration Delay
- Feature that allows **in-flight (ongoing) requests to complete**
  before an EC2 instance is removed from a Load Balancer.
- Prevents **abrupt connection termination**.

## 8.1 Naming by Load Balancer

| Load Balancer | Name |
|--------------|------|
| **CLB (Classic)** | Connection Draining |
| **ALB (Application)** | Deregistration Delay |
| **NLB (Network)** | Deregistration Delay |

## 8.2 How It Works

1. Instance is:
   - Deregistered OR
   - Marked unhealthy
2. Load Balancer:
   - **Stops sending NEW requests** to that instance
3. Existing users:
   - Allowed to **finish their current requests**
4. After timeout:
   - Connections are **terminated**
## 8.3 Configuration

- **Range:** 1 → 3600 seconds  
- **Default:** 300 seconds (5 minutes)  
- **0 seconds:** Disabled (immediate termination)

## 8.4 Behavior Summary

- ✅ Existing connections → allowed to finish  
- ❌ New connections → NOT sent to draining instance  
- ⏱ After timeout → instance fully removed

## 9. Health Checks

### Purpose
- Determine if an instance is **healthy or unhealthy**  

### Configuration
- Protocol (HTTP / HTTPS / TCP)  
- Port  
- Path (e.g., `/health`)  

### Behavior
- Returns **200 OK → Healthy**  
- Otherwise → Unhealthy  

### Result
- Unhealthy instances **do NOT receive traffic**

## 10. Types of Load Balancers

### 10.1 Classic Load Balancer (CLB)
- Legacy (2009)  
- Supports HTTP, HTTPS, TCP, SSL  
- **Not recommended**  

---

### 10.2 Application Load Balancer (ALB)

- Layer 7 (Application layer)  
- Supports:
  - HTTP, HTTPS  
  - WebSocket  

### Features
- Path-based routing (`/api`, `/images`)  
- Host-based routing (`app.example.com`)  

### Use Case
- Web apps  
- Microservices  

---

### 10.3 Network Load Balancer (NLB)

- Layer 4 (Transport layer)  
- Supports:
  - TCP, TLS, UDP  

### Features
- Very high performance  
- Low latency  
- Handles millions of requests  

### Use Case
- Real-time apps  
- Gaming  
- High-performance systems  

---

### 10.4 Gateway Load Balancer (GWLB)

- Layer 3 (Network layer)  
- Used for:
  - Firewalls  
  - Security appliances  
  - Traffic inspection  

---

## 11. Internal vs External Load Balancer

### Internet-Facing (Public)
- Accessible from internet  
- Used for public applications  

### Internal (Private)
- Accessible only within VPC  
- Used for internal services  

---

## 12. Security

### Load Balancer Security Group
- Allows traffic from users  
- Example:
  - Port 80 / 443  
  - Source: `0.0.0.0/0`  

---

### EC2 Security Group
- Allows traffic **ONLY from Load Balancer**  
- Source:
  - Load Balancer Security Group  
### Why?
- Prevent direct access to EC2  
- Forces all traffic through Load Balancer  

---

## 13. Key Concepts Summary

- Load Balancer = **traffic distributor + single endpoint**  
- Improves:
  - Scalability  
  - Availability  
  - Fault tolerance  

---

## 14. Key Points
- **ALB → HTTP/HTTPS (Layer 7)**  
- **NLB → TCP/UDP (Layer 4, high performance)**  
- **GWLB → security appliances**  
- **CLB → legacy (avoid)**  

- Health checks:
  - If instance fails → **no traffic sent**  
- Security:
  - EC2 should **only allow traffic from LB**  
- Choose:
  - **ALB → web apps**
  - **NLB → performance**
